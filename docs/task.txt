Проект: ИИ агентная система для автоматизации анализа событий ИБ.
Задачи системы:
1. Сбор алертов из SIEM
2. Категоризация алертов по типу инциидентов (например брутфорс, эксплуатация уязвимости в веб-сервисе)
3. Поиск информации, плейбуков в базе знвний по обработке конкретных типов инцидентов
4. Создание четкого пошагового плана по анализу инцидета, его обогащению информацией
5. Отправка плана аналитику.

В качестве тестовой среды для проверки работоспособности можно использовать SIEM Wazuh

Документация Wazuh - https://documentation.wazuh.com/current/index.html

Для работы с LLM можно использовать внешние api (например openrouter) и локальные, например ollama

Работу с данными нужно реализовать внутри SIEM. То есть, для реализации последнего модуля с отправкой плана нужно сделать так, чтобы он выводился внутри SIEM через отдельный дашборд, а не отдельный веб-интерфейс систему.

Ниже представлено мое видение реализации данных модулей, данной структуры нужно придерживаться.

3.2 Коннектор
Коннектор - это модуль, отвечающий за взаимодействие агентной системы со средствами мониторинга, в частности с SIEM. Он представляет из себя отдельный микросервис, который подключается к средствам мониторинга и агентной системе через API и позволяет перенаправлять события безопасности в агента. Данный подход позволяет легко адаптировать работу системы под разные средства мониторинга, не меняя архитектуру компонентов  самой агентной системы.
Основные задачи коннектора следующие:
1. Взаимодействие с SIEM через API, получение из них событий информационной безопасности.
2. Первичная нормализация событий из SIEM  и приведение их в единый формат.
3. Управление очередью событий и передача нормализованных данных в агентную систему
Получение данных из SIEM реализуется двумя способами, дополняющими друг друга. Первый - использует REST API для запросов к SIEM. В данном сценарии запрос на получение данных инициализируется клиентом (агентной системой) и включается по срабатыванию условия, времени или по ручному запросу. Это подходит для получения дополнительного контекста в ходе анализа алерта, над которым уже ведется работа. Второй сценарий использует вебхуки (webhook) для отправки данных из SIEM в агента при достижении условия, например, при создании нового оповещения безопасности. Здесь клиент (агент) является слушателем  и ждет, когда ему отправят данные для анализа, что обеспечивает минимальное время реакции системы на новые события безопасности. Комбинирование обоих подходов позволяет достичь хорошей скорости реагирования и качества анализа.
Следующий важный этап - это нормализация событий и приведение их к единому унифицированному формату, что важно для качественного анализа учитывая то, что разные SIEM-системы (Splunk, MaxPatrol SIEM, KUMA и другие) используют собственные форматы представления событий. Для стандартизации существуют общепринятые форматы такие как Common Event Format (CEF), Log Event Extended Format (LEEF) и Elastic Common Schema (ECS). В рамках проектируемой системы коннектор выполняет нормализацию событий в формат ECS.
Ключевым элементом коннектора является брокер сообщений и система очередей. Это программное обеспечение, которое принимает сообщения от одного сервиса и передаёт их одному или нескольким другим, обеспечивая гарантированную доставку и буферизацию при пиковых нагрузках. Среди реализаций подобных систем выделяются Apache Kafka и RabbitMQ. Необходимость использования данного программного обеспечения обусловлена ограниченностью вычислительных ресурсов системы, которая использует большие языковые модели и тем, что средства мониторинга создают непрерывный поток большого количества события. В данном сценарии алерты сначала поступают в брокер и становятся в очередь, а агентная система извлекает их по мере завершения предыдущей обработки.



3.3 Модуль планирования
Модуль планирования отвечает за категоризацию поступивших из SIEM событий и формирование плана дальнейшего анализа. Он представляет из себя систему из двух языковых моделей, первая из которых обладает небольшим числом параметров и используется для определения категории предполагаемого инцидента. Она способна обрабатывать маленький объем информации, но с высокой скоростью. Вторая – большая языковая модель, способная обрабатывать большой объем информации и создавать осмысленный текст. Она используется для написания плана анализа инцидента. Для работы с актуальной информацией используется технология RAG (генерация с дополненной выборкой) — подход, при котором ответ большой языковой модели создается на основе данных, полученных из  внешних источников [31]. Исследование [35] показало, что интеграция механизма поиска в процесс генерации позволяет существенно повысить фактическую точность ответов и снизить частоту галлюцинаций по сравнению с использованием только параметрических знаний модели.
Схема работы модуля заключается в следующем:
На вход модуля подается алерт от SIEM системы, полученный через коннектор и отправляется в первую небольшую языковую модель для категоризации.
Модель анализирует данные и на основе своих знаний определяет тип предполагаемого инцидента.
После категоризации делается запрос к базе знаний через RAG для поиска информации по обнаружению и обработке конкретного типа инцидента.
После получения нужных данных запрос передается в большую языковую модель для формирования структурированного плана анализа и шагов действий для обнаружения инцидента.
Сформированный план вместе с исходным событием передаётся аналитику
